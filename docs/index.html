The High-Performance Engine.

Reproq Worker is a multi-threaded execution engine written in Go. It handles the heavy lifting of polling, heartbeat management, and process execution.

Low Latency

Atomic polling with SKIP LOCKED ensures sub-millisecond task pickup.

Self-Healing

Built-in lease reaper and heartbeat system prevents zombie tasks.

Why Reproq

Postgres Only

No Redis/RabbitMQ. One datastore, one worker.

Self-Healing

Leases expire and reclaim automatically when workers die.

If you need complex routing or multi-broker support, Celery may still fit better. Reproq optimizes for determinism and low operational overhead.

Installation

Reproq provides pre-built binaries for Linux, macOS, and Windows. You can download them manually or use the Django helper.

Quickstart (Django)

python manage.py reproq install
python manage.py reproq migrate-worker
python manage.py reproq worker

Manual Install (Go)

go build -o reproq ./cmd/reproq

Cross-Platform Binaries

Download from the Releases page. Binaries follow the pattern reproq-{os}-{arch}.

Schema note: Reproq Worker expects JSONB columns for task_runs.worker_ids and reproq_workers.queues. If you previously applied legacy SQL migrations that created array columns, run migrations/000013_convert_worker_arrays_to_jsonb.up.sql from the reproq-worker repo.

Command Reference

worker

Starts the execution node.

./reproq worker --dsn $DB_URL --concurrency 20 --metrics-port 9090

beat

Starts the cron scheduler.

./reproq beat --dsn $DB_URL --interval 30s

replay

Re-enqueues a task.

./reproq replay --id 12345

limit

Manage rate limits (token buckets) in Postgres.

./reproq limit set --key queue:default --rate 5 --burst 10
./reproq limit ls
./reproq limit rm --key queue:default

Advanced Features

Global Rate Limiting

Token bucket enforcement is built in but disabled by default. Configure queue or task limits via reproq limit or the database.

Priority Aging

Dynamic priority prevents low-priority tasks from starving. Older tasks gain effective priority based on PRIORITY_AGING_FACTOR.

Deterministic Deduplication

Built-in spec hashing ensures that identical tasks are coalesced rather than duplicated, maintaining strict idempotency.

Self-Healing Leases

Tasks are claimed with a heartbeat-backed lease. If a worker process dies, the system automatically returns the task to the queue after the lease expires.

Prometheus Monitoring

Enable the built-in metrics exporter to monitor throughput and error rates.

# Start with metrics enabled
./reproq worker --metrics-port 9090

Exposed metrics include:

  • reproq_tasks_processed_total: Counter of tasks by queue and status.
  • reproq_task_duration_seconds: Histogram of execution times.
  • Standard Go runtime metrics (goroutines, heap, etc).

Deployment

For production deployment, it is recommended to use systemd to manage the worker and beat processes. Reproq provides an automated command to generate these service files for you:

python manage.py reproq systemd

See the Deployment Guide for manual templates and best practices.

Note: Reproq Worker is designed to work in tandem with Reproq Django, which provides the task definition and monitoring interface.

Scaling: Workers vs Concurrency

You can scale Reproq by increasing --concurrency or by running multiple workers.

  • Concurrency: more goroutines in a single process; best for I/O-bound workloads with minimal overhead.
  • Multiple workers: separate processes or hosts; best for CPU-bound workloads and fault isolation.

Rule of thumb: start with 1-2 workers per host and tune --concurrency to your CPU cores and workload type.
